# 設計書：Stefnceorf — テキスト編集で音声を編集するCLIツール v2

正式名: Stefnceorf / コマンド: `stefnceorf` / 2文字エイリアス: `sc`

作成日: 2026-07-20 / 状態: 実装用（Claude Code入力向け）
v1からの変更：文字起こしをmlx-whisperに変更（導入済み環境の流用）、フィラー語の自動削除提案を追加

## 1. 目的

音声（沈黙除去・ノイズ除去済みのwav）を文字起こしし、そのテキストファイルを編集（削除・並べ替え）すると、編集結果が音声ファイルに反映されるCLIツールを作る。Descriptの音声編集機能の日本語対応・ローカル・無料版に相当。

## 2. スコープ

やること：

- 日本語・英語の文字起こし（単語単位タイムスタンプ＋単語ごとの信頼度）
- テキストの「削除」「行（セグメント）単位の並べ替え」を音声へ反映
- フィラー語（あのー、まあ、えー等）の自動検出と削除提案
- 低信頼箇所のマーキング（聞き直し箇所の絞り込み）
- 全処理ローカル・無料

やらないこと：

- 動画、テロップ出力（将来拡張。§11）
- 加筆・言い換え（元音声が存在しないため不可）
- GUI
- 沈黙・ノイズ除去（Audacityで前処理済みの前提）

## 3. 全体構成

```
input.wav（元音質のまま保持、読み取り専用）
   │  stefnceorf transcribe
   │    ffmpegで16kHz mono一時wavに変換 → mlx-whisperへ（既存運用と同じ）
   ▼
input.sc.txt      ← 人間が編集するテキスト（行=セグメント、行頭にID）
input.sc.json     ← 単語→時刻・信頼度の対応表（触らない）
   │  （エディタで削除・行の並べ替え・フィラー提案の採否）
   ▼
   │  stefnceorf render   ← 切り出しは「元のinput.wav」から行う（音質劣化なし）
   ▼
input.edited.wav
```

## 4. CLI仕様

```
stefnceorf transcribe input.wav [--lang ja|en] [--model MODEL] [--filler-suggest]
  → input.sc.txt / input.sc.json を生成
  --model 既定: mlx-community/whisper-large-v3-turbo（導入済み）

stefnceorf render input.sc.txt [-o output.wav]
  → 編集後テキストとjsonを突き合わせて音声を再構成
```

2コマンドのみ。エイリアス `sc` を同梱し `sc transcribe` / `sc render` でも同一動作。renderは何度でも再実行可（非破壊）。実装はPython、mlx_whisperはCLIではなくPython API（`mlx_whisper.transcribe(..., word_timestamps=True)`）で呼ぶ。transcribeオプションは既存シェルスクリプトの実績値を踏襲（temperature=0, condition_on_previous_text=False, no_speech_threshold=0.8, compression_ratio_threshold=2.0）。

## 5. 編集ファイル仕様（input.sc.txt）

```
[0001] 結局面倒くさい作業があるんだったらツールを作ってしまえばいいの
[0002] 〔まあ〕とか〔その〕とかそういう言葉を一個ずつ消して
[0003] 私ですね〔あのー〕動画で喋りたいこととかはいくらでも◆あるんですよ
```

- 1行 = 1セグメント。行頭 `[ID]` が対応表へのキー
- **行削除** → セグメント全体が音声から消える
- **行の並べ替え** → その順で音声が再構成される
- **行内の文字削除** → 対応する単語の音声が消える
- **`〔...〕`（フィラー提案）** → renderが既定で削除する。残したい場合は括弧だけ消す（例：`〔まあ〕`→`まあ`）
- **`◆`（低信頼マーカー）** → 信頼度スコア閾値以下の単語の直前に付与。該当箇所だけ聞いて確認する。renderは無視するので消さなくてもよい
- IDのない行・不明なIDはrenderがエラーで報告（黙って無視しない）

制約：

- 行内での文字の「追加・書き換え」は無視して警告（削除のみ有効）
- 並べ替えは行単位のみ（行内語順の入れ替えは非対応）

## 6. フィラー語の自動削除提案

- 方式：辞書照合。Whisperの単語トークンが辞書と**完全一致**した場合のみ `〔〕` で包む（部分一致はしない）
- 既定辞書（ja）：あの, あのー, その, そのー, えー, え, えっと, えと, まあ, まー, なんか, こう
- 既定辞書（en）：um, uh, uhm, er, ah
- 辞書は設定ファイル（例：`fillers_ja.txt`、1行1語）で編集可能
- **あくまで提案**。指示語の「その」「あの」（例：そのウェブサイト）と間投詞は音声認識結果だけでは区別できないため、誤検出は起こる。だからこそ自動確定ではなく `〔〕` 提案とし、ユーザーが括弧を外して却下できる設計にする
- transcribe時に検出数を表示（例：`フィラー候補: 37箇所`）。フィラー提案は既定で無効。`--filler-suggest` で有効化（既定は無効）

## 7. 技術選定

| 役割 | 採用 | 理由 |
|---|---|---|
| 文字起こし＋単語時刻＋信頼度 | mlx-whisper（large-v3-turbo, word_timestamps=True） | 導入済み・Apple GPUで高速。wordごとにstart/end/probabilityが取れるため1モデルで完結 |
| 行内diff | Python difflib（文字単位） | 日本語は分かち書きがないため文字単位で照合し、単語へ逆引き |
| 前処理変換 | ffmpeg（16kHz mono一時wav） | 既存スクリプトと同一。認識用のみ。切り出しは元wavから |
| 音声再構成 | numpy + soundfile | サンプル配列で切り貼り。カット境界に5–10msフェードでクリックノイズ防止 |

v1（faster-whisper＋WhisperX 2段構成）からの変更理由：導入済み資産の流用と依存の削減。トレードオフとして、Whisper自前の単語タイムスタンプは強制整列（WhisperX）より境界精度が劣る可能性がある。マージン＋クロスフェードで吸収を狙い、実音声で語切れが目立つ場合のみWhisperXを整列段として追加する（§11）。

## 8. renderのロジック

1. 編集後txtを行単位でパース → ID列（＝出力順）
2. `〔...〕` が残っている箇所は削除扱いに変換してからdiffへ
3. 各行：編集後文字列 vs 元セグメント文字列を文字diff → 生存文字 → 対応表で「残す単語」へ
4. 連続単語をマージして「残す時間区間」リスト化（前後20ms程度のマージンで境界誤差を吸収。ただし隣接カット区間に食い込まない範囲で）
5. 元wavから区間を切り出し、境界クロスフェードで結合 → 書き出し（元wavと同一のサンプルレート・ビット深度）

## 9. 精度と限界（事実ベース）

- 日本語の文字起こし精度は93–95%程度が相場。固有名詞・専門用語がブレやすい
- 単語タイムスタンプ誤差で語尾欠け・隣音混入がありうる → マージン＋フェードで大半緩和。最終の通し聞き1回は前提
- フィラー誤検出（指示語との混同）はレビューで却下する運用
- turboモデルはlarge-v3比でわずかに精度が落ちる報告あり。気になる場合は `--model mlx-community/whisper-large-v3-mlx` に切り替え可能な設計にする

## 10. 没にした案（記録）

- **faster-whisper＋WhisperX 2段構成（v1案）**：境界精度は最良だが、依存が重く導入済み資産も使えない。v2では保留し、必要になったら整列段のみ追加
- **プレーンテキスト＋fuzzy diffのみ（ID行なし）**：並べ替え対応でdiffの曖昧性が跳ね上がるため却下
- **ffmpeg filter_complexで結合**：カット数百でコマンド破綻しがち。numpy直操作の方が単純
- **クラウドAPI**：ローカル・無料の要件に反する
- **フィラーの自動確定削除**：誤検出（指示語）を黙って消すのは危険。提案方式に変更

## 11.5 実装後の仕様変更（2026-07-21、ユーザー承認済み）

- **無音切り詰めをスコープに追加**（§2の「沈黙除去はやらない」を変更）：前処理はノイズ除去のみでよい。transcribeは認識用一時wavの無音（1.5s超）を0.7sに切り詰めてWhisper幻覚を抑止し、単語時刻を元音源へ逆写像。renderは残存区間間の無音ギャップを保持しつつ1.5s超を0.7sに切り詰め（`--gap-threshold` / `--gap-max`）
- **.sc.txt 行頭に開始時刻を併記**：`[0001 12:34] テキスト`（renderは時刻を無視、旧形式互換）
- **フィラー既定辞書（ja）を間投詞のみに縮小**：指示語系（あの・その・なんか・こう）を除外し誤検出を削減
- **フィラー提案を既定無効に変更**：日本語ではWhisperがフィラーを吸収するため検出率が低い。`--filler-suggest` で有効化（既定は無効）
- **ポーズベース区切り（ブロック単位削除）を追加**（2026-07-21、ユーザー承認済み）：
  - 課題：削除は従来「単語単位スナップ」だが、音がつながった箇所で切ると結合が不自然。単語タイムスタンプはDTWが隙間を埋めるためポーズ判定に使えない（実データ55分でセグメント内ギャップ≥0.1sは13箇所のみ）
  - 方式：transcribeで ffmpeg silencedetect を1回だけ実行し、`d=min(pause_threshold, 1.5)` で検出。結果を用途別にフィルタ（切り詰め用＝duration≥1.5s／ブロック境界用＝duration≥`pause_threshold`）。純関数 `assign_blocks` が各無音の中点を最寄りの単語接合点にマップし、単語ギャップ≥閾値も境界として、各 word にセグメント内 `"block"`（0始まり）を付与。jsonトップに `"pause_threshold"` を保存
  - 表示：`.sc.txt` のブロック境界に区切り記号 `／`（全角 U+FF0F、実データ transcript に出現0件）を挿入。位置は 〔〕の外側・◆の前・lead空白より前。◆〔〕／を除去すると word 連結に一致する不変条件を維持
  - render：純関数 `snap_to_blocks` で「ブロック内に1語でも削除があればブロック全体を削除」。`surviving_words`（文字→単語）はそのまま、後段に単語→ブロックの第2スナップを追加。巻き込み削除は警告で列挙。`_clean_edited_text` が diff 前に `／` を除去するため、`／` を残す/消すは等価
  - 後方互換：`block` を持たない word があるセグメントは `blocks=[0,1,2,...]`（各単語独立＝従来の単語スナップ）にフォールバック。旧 .sc.json/.sc.txt は無修正で動く
  - 逃げ道：`--pause-threshold 0` で全単語境界＝旧挙動（単語単位削除）
- **verbatim モード（フィラーも転写）を追加**（2026-07-22、ユーザー承認済み）：
  - 課題：既定設定（large-v3-turbo, `condition_on_previous_text=False`, プロンプトなし）では Whisper がフィラーを吸収し転写されない。フィラー削除ワークフローの前段としてフィラーを転写させたい
  - 実験結果：`mlx-community/whisper-large-v3-mlx` ＋ initial_prompt（フィラー例文）＋ `condition_on_previous_text=True` でフィラー（「まあ、」「あの、」「えっと」等、読点付きトークン）が転写されることを確認。turbo+condF はフィラー吸収、turbo+condT は繰り返し幻覚が出るため NG、large+prompt+condT では幻覚なし
  - 方式：`transcribe(..., verbatim=False)` を追加。`--verbatim` 指定時、`--model` 未指定なら `VERBATIM_MODEL`（large-v3-mlx）に差し替え（明示モデルは尊重するため `model` を sentinel `None` 方式に変更）、`initial_prompt`（日本語 `FILLER_PROMPT` / 英語 `FILLER_PROMPT_EN`）と `condition_on_previous_text=True` を渡す。非 verbatim 時は現行どおりプロンプトなし・False。幻覚対策の既存パラメータ（temperature=0, compression_ratio_threshold=2.0, no_speech_threshold=0.8）は不変。json トップに `"verbatim"` を記録
  - フィラー辞書：verbatim 転写は読点付き（「まあ、」）で出るため、`normalize_token` を前後の句読点・記号除去に拡張（長音「ー」は残す）。指示語系（あの・その・なんか・こう）は `SOFT_FILLERS_JA` として、元トークンが読点/句点で終わる場合のみフィラー扱い（「あの、」は拾い「あの本」は誤検出しない）
  - `--filler-suggest` の自動有効化はしない（独立のまま。併用を README で推奨）
- **繰り返し幻覚セグメントの後処理除去を追加**（2026-07-22、ユーザー承認済み）：
  - 課題：`--verbatim`（`condition_on_previous_text=True`）で長尺ファイルの静音区間から繰り返し幻覚が発生する。実データ（55分）では「今、」だけのセグメントが数十行連続／1セグメント内に「今、」トークンが100個以上詰まった巨大セグメント／words が空のセグメントが約50行連続、といった形で現れた
  - なぜ後処理か：condT の幻覚は initial_prompt・temperature・compression_ratio_threshold 等の入力側パラメータでは完全には防げない（condT を切ると今度はフィラーが吸収され verbatim の目的を損なう）。そのため認識結果に対する後処理として、パターンが明確な幻覚のみを保守的に除去する
  - 方式：純関数 `drop_hallucinations(segments) -> (kept, dropped)` を追加。remap 後・ID採番前の生セグメント列に対し、セグメントループの前に適用する。検出条件は3種（いずれか該当で除去）：
    - a. **セグメント内反復**：空白除去した単語トークン（`w["word"].strip()`）が `HALLUC_MIN_WORDS`(=5) 個以上あり、最頻トークンの割合が `HALLUC_TOP_RATIO`(=0.7) 以上（巨大反復セグメント対策）
    - b. **セグメント列反復**：`text.strip()` が同一の非空セグメントが `HALLUC_RUN`(=3) 個以上連続する場合、その連続全部を除去（交互パターン「ああして、/こうして、」は同一連続でないため残す＝正常な繰り返し表現を巻き込まない）
    - c. **空セグメント**：`words` が空、または `text.strip()` が空
  - 記録：除去があれば json トップに `"hallucination_drop": {"count": N, "ranges": [{"start", "end", "sample"}]}` を追加。ranges は連続する除去を1範囲にまとめた元音源時刻（`rec_to_src(..., cuts)` で認識用wav時刻から逆写像）。CLI は stderr に「除去数＋各区間（M:SS 形式）＋先頭サンプル」を警告表示する
  - 影響：除去区間は文字起こしが無くなるため render で音声も削除される。ゆえに警告で聞き直しを促す。残ったセグメントの ID は詰めて連番採番する（0001,0002,...）
- **幻覚区間の再認識レスキューを追加**（2026-07-22、ユーザー承認済み）：
  - 課題：実データで幻覚除去区間（19:43–21:33 の約2分）に実発話が存在することが判明。`drop_hallucinations` の除去だけでは発話が失われる。同区間を安全設定（`condition_on_previous_text=False`・プロンプトなし）で個別に再認識すると完全に正しく転写できることを実証済み
  - 方式：`drop_hallucinations` はそのまま使い、その除去グループごとに**認識用wav時刻**でレスキュー窓を決めて安全設定で再認識し、生き残ったセグメントを kept 列の該当位置に差し替え挿入する。認識用wav（`trimmed_wav`）は本認識直後に削除していたが、レスキュー完了まで生存させるよう本認識〜レスキューを1つの `try/finally` に再構成した（一時ファイル削除漏れ防止）
  - レスキュー窓（純関数 `rescue_window(prev_kept_end, next_kept_start, wav_duration)`）：窓 = 「直前の kept セグメント末尾単語の end」〜「直後の kept セグメント先頭単語の start」（認識用wav時刻）。先頭グループは 0、末尾グループは認識wav長を境界に使う。窓長 < `RESCUE_MIN_WINDOW_S`(=0.5s) なら None（内容なしとみなし従来通り除去扱い）
  - **なぜ窓を kept 境界から取るか**：(1) kept セグメントと**重複しない**（差し替え時に二重転写を防ぐ）、(2) 除去グループに時刻を持たない**空セグメントが含まれても失われた音声を全部カバーできる**（空セグメントの窓は前後 kept 境界で確定するため）
  - 再認識：`_slice_wav` が `soundfile` で窓のサンプルを一時wavに切り出し、`mlx_whisper.transcribe` を安全設定（`word_timestamps=True, language=lang, temperature=0, condition_on_previous_text=False, no_speech_threshold=0.8, compression_ratio_threshold=2.0`, `initial_prompt` なし）で実行。得られた単語 start/end に窓開始時刻を加算して認識用wav時刻へ戻し、レスキュー結果にも `drop_hallucinations` を1回だけ適用（ガード。再帰レスキューはしない）
  - 記録：`hallucination_drop.ranges` の各要素を `{"start", "end", "sample", "rescued": bool, "rescued_segments": N}` に拡張。`count` は本認識で幻覚検出した除去数（レスキュー成否によらない）。時刻は元音源時刻（除去単語時刻を優先、無ければ窓境界を `rec_to_src` で逆写像）
  - CLI：`rescued` な区間は「幻覚疑い区間を再認識で復旧: M:SS-M:SS（N セグメント）」と stdout に通常表示。復旧できなかった区間のみ従来の警告（stderr、聞き直し案内）
  - 影響：復旧区間は安全設定（verbatim でない）で認識されるためフィラーは転写されない
- **verbatim フィラー削除の精密化（無音クランプ・ID構造マッチ・音響安全カット）を追加**（2026-07-22、ユーザー承認済み）：
  - 動機：verbatim ＋ `〔〕` 削除の実運用で3つの不満が出た。(1) **巻き込み**：頻出フィラーは文字diffの整列が曖昧になり隣接語まで削除・ブロック巻き込みが起きる、(2) **削除漏れ**：フィラー単語のタイムスタンプが実発話より後ろへずれ（実測 end 2.30s vs 実際の無音開始 1.20s）カット位置がずれる、(3) **不自然**：前後の音と連続調音しているフィラーを機械的に切ると結合が不自然になる
  - 採用した設計：
    - **無音クランプ**（`clamp_words_to_silences`、`CLAMP_MIN_WORD_S=0.05`）：transcribe で検出済み無音に食い込む単語境界を無音の外へ縮め、フィラー等の時刻ズレを補正（stable-ts の `adjust_by_silence` 相当を自前実装）。`.sc.json` に `silences`（`pause_threshold` 以上のポーズ）を保存
    - **ID 構造マッチ**（`match_filler_deletions`）：`〔〕` 提案単語に `suggest` フラグを付けて `.sc.json` に保存。編集後に残った `〔中身〕` を提案単語へ出現順マッチし、対応 word ID を diff の両辺から除外して削除確定。文字diffの整列曖昧性に依存せず巻き込み・削除漏れを根絶（残す＝削除指定という意味付けを ID で確定）
    - **スナップ除外＋音響安全判定**（`snap_with_filler_exemption` / `filler_cut_is_safe`）：フィラー削除確定分はブロックスナップの巻き込み対象から除外して精密カット。ただしカット両境界が無音に接しない（silences 端 ±10ms でなく、かつ境界 ±`RMS_SAFE_WINDOW_S=0.03` の RMS が `RMS_SAFE_DBFS=-40` を超える＝発話が連続している）フィラーは自然さ優先でカットせず keep に戻し警告。手動で文字を消したフィラーも同ブロックに残存語があれば同じ精密カット対象に取り込む
    - **片側ポーズ残し**（`plan_output_intervals` のフィラー分岐、`FILLER_PAUSE_KEEP_S=0.25`）：フィラー削除後は前後の無音から合計約0.25秒の間を残して単純結合（無音中のカットのためクロスフェードしない）
    - **単語内無音ガード**：無音区間が単一単語の内部に完全に収まる場合はブロック境界（`／`）を作らない（単語途中での不要なカット単位分割を防ぐ）
  - 却下した代替：外部ライブラリの A/B テストを実施し不採用。**stable-ts** は有効なのは無音補正のみで自前実装（無音クランプ）で代替可能、依存追加に見合わない。**WhisperX** は幻覚語の強制配置（アライメントが実在しない語を無理に置く）とフィラーの誤配置が起きたため不採用
  - 主要定数：`CLAMP_MIN_WORD_S=0.05`, `FILLER_PAUSE_KEEP_S=0.25`, `RMS_SAFE_DBFS=-40`, `RMS_SAFE_WINDOW_S=0.03`
  - **谷スナップ＋相対RMS閾値を追加**（2026-07-22 実データ検証後の改良）：絶対閾値 -40dB が実録音の発話帯域（中央値 -38dB、確実な無音は -73dB 帯）とほぼ重なり、フィラー 67 件中 49 件が誤って unsafe 判定された。対策として (1) `plan_filler_cut` / `snap_to_trough`：カット両境界を ±`SNAP_SEARCH_S=0.10` 秒内の RMS 最小点（無音の谷）へスナップ（隣接語に食い込まないよう前語 end / 次語 start でクランプ、スナップ後 `MIN_FILLER_CUT_S=0.03` 未満はカットしない）、(2) `speech_median_dbfs`：閾値を `min(RMS_SAFE_DBFS, 発話中央値 - RMS_SAFE_REL_DB=15)` に相対化（録音レベルへ自動適応しつつ -40dB より緩くしない）。ep22-1 実測で切除率 27%→76%（67 件中 51 件カット・16 件は谷でも発話帯域＝真に連続調音のため自然さ優先で残す）。ep22-2 でも RMS 分布の再現を確認
  - 後方互換：`silences` / `suggest` / `block` を持たない旧 `.sc.json` でも動く。`suggest` 無しは `is_filler` フォールバック、`silences` 無しは音響安全判定を RMS のみへ劣化、`block` 無しは各単語独立ブロック
  - CLI：`sc trans` を `sc transcribe` のエイリアスとして追加
- **幻覚抑止の強化（silence_threshold 常時付与・密度条件d・エコー条件e）を追加**（2026-07-22、ユーザー承認済み）：
  - 課題：`--verbatim`（`condition_on_previous_text=True` ＋ large-v3）で長い静音起点の繰り返し幻覚が既存の条件a/bをすり抜けた。実データ（260720_ep22-2）で、直前セグメントとほぼ同文だが完全一致でない繰り返しが 32文字/0.12秒（**266文字/秒**、全単語ほぼ zero-duration）という異常密度で出た。実発話は 5〜7文字/秒。条件b（同一テキスト `HALLUC_RUN`=3 連続）は 2連＋前後の微差で捕まらなかった
  - 対策1（入力側）：`hallucination_silence_threshold`（mlx-whisper 0.4.3 実装済みだが未使用。幻覚疑い時に無音をスキップする本家 whisper 由来の抑止、`word_timestamps=True` 前提＝当ツールは常に満たす）を定数 `HALLUC_SILENCE_SKIP_S=2.0` で本認識・レスキュー再認識の双方へ**常時付与**（非verbatim・condT=False でも幻覚は起き得るため）。2.0s は本家推奨域でポーズ区切り 0.3s より十分長く発話間ポーズを誤スキップしない
  - 対策2（後処理・条件d 時間密度異常）：`drop_hallucinations` に条件dを追加。トークンが `HALLUC_MIN_WORDS`(=5) 個以上で、発話時間（先頭単語 start〜末尾単語 end）が正なら文字密度 = `len("".join(tokens))/duration` が `HALLUC_DENSITY_CHARS_S`(=25.0、実発話5〜7文字/秒の3倍超) 超で除去、発話時間 ≤0（全単語 zero-duration）でも除去。条件aに該当しなかった場合に評価
  - 対策3（後処理・条件e エコー）：セグメントの正規化テキスト（空白と句読点「、。，．！？!?…・,. 」を除去）と**生の直前セグメント**（条件dで除去済みでも可＝幻覚列の後続が除去済み幻覚のエコーであるケースを捕まえる）の共通接頭辞が `HALLUC_ECHO_PREFIX`(=15) 文字以上、かつ文字密度が `HALLUC_ECHO_DENSITY_CHARS_S`(=12.0) 超で除去。幻覚エコーは直前文の微変形（前置き欠落・末尾延長）で出るため完全一致の条件bでは捕まらない。実発話の言い直し反復もセグメント全体が12文字/秒超にはならないため密度条件で誤除去を防ぐ
  - 評価順：c → a → d →（別ループ）e →（最後）b

## 12. 受け入れテスト

1. **往復テスト**：transcribe→無編集でrender→出力が元とほぼ同尺・内容同一
2. **行削除**：1行消してrender→該当発話だけが消えている
3. **行内削除**：文中の数文字を消してrender→対応する単語だけが消え、前後が自然につながる
4. **並べ替え**：2行を入れ替えてrender→音声順序が入れ替わる
5. **フィラー**：`〔まあ〕`をそのままrender→消える。括弧を外してrender→残る
6. **異常系**：ID行破損・不明IDでrenderが明確なエラーを出す
7. **音質**：出力が元wavと同一フォーマット、カット境界にクリックノイズがないこと（波形目視＋試聴）
